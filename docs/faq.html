General

Product basics.

What does CSP Automation Manager do?

It helps WordPress administrators build, test, and enforce Content Security Policy headers. It emits per-surface policies, injects nonces, collects browser reports, discovers sources, and requires administrator approval before new sources affect the policy.

Who is it for?

It is for site owners, agencies, managed service providers, and WordPress administrators who need a safer path from no CSP or weak CSP to a stricter policy.

Does it require deep CSP knowledge?

No, but users still need to understand the basic workflow: report-only first, collect evidence, approve required sources, reject unknown sources, then enforce one surface at a time.

What are surfaces?

Surfaces are separate parts of the site. The plugin uses frontend, admin, login, and API. Each surface can have its own mode and policy inventory.

Why not use one policy for the whole site?

Different parts of WordPress need different resources. A frontend page, login page, REST response, and admin screen do not have the same risk profile or resource requirements.

What are the minimum requirements?

WordPress 6.4 or later and PHP 8.1 or later.

Rollout

Safe deployment questions.

Will report-only mode block anything?

No. Report-only mode asks the browser to report what would have been blocked. It does not block the resource.

Can enforce mode break a site?

Yes. Enforce mode can block required scripts, styles, fonts, frames, media, form posts, and runtime API calls. Use enforce mode only after reviewing evidence.

Which surface should I enforce first?

Usually the frontend surface, after testing key public journeys. Login, API, and admin surfaces should follow only when there is enough evidence.

Should I enforce wp-admin?

Use caution. WordPress admin screens and plugins may emit inline code outside the normal script API. Keep a rollback path and monitor violations carefully.

What is the promotion gate?

It is a safety check before enforce mode. The surface must have approved source or hash inventory, must not have recent violations inside the configured window, and must not have an active temporary override.

Why is the enforce button blocked?

The promotion gate has failed. Check approved inventory, recent violations, and temporary override status for the surface.

How long should I stay in report-only mode?

Long enough to test real journeys and pass the configured violation-free window. For production sites, start with at least 24 hours. Complex sites may need longer.

Approvals

Source inventory and policy decisions.

Should I approve every discovered source?

No. Approve only sources that are necessary, expected, and narrow. Reject or investigate anything unknown.

What does high risk mean?

High-risk proposals usually affect script, style, connection, form, frame, or worker behaviour, or use broad patterns such as wildcards, cleartext HTTP, broad schemes, or unsafe CSP keywords.

Is a high-risk source always bad?

No. A payment provider, consent manager, or known application endpoint may be legitimate. High risk means the proposal deserves deliberate review.

What is evidence count?

It is the number of times the same source has been observed or refreshed. It can help distinguish recurring site behaviour from one-off noise.

What happens when I reject a source?

The decision is recorded and the matching surface, directive, and host fingerprint is suppressed from future automatic proposals unless a later approval becomes the newest decision.

What is the difference between reject and revert?

Reject is for a pending or denied source that should not be approved. Revert is for a previously approved source that should be withdrawn.

Why did a rejected source appear again?

Check whether the directive, host, or surface changed. Suppression is based on the source fingerprint, not every possible variant of a service.

Violations

Browser reports and learning behaviour.

Where are violation reports stored?

They are stored locally in the WordPress database in the plugin violation reports table. Old reports are purged by the daily scan according to the configured retention period. The default is 90 days; zero keeps reports indefinitely.

Are reports sent to a third-party reporting service?

No third-party reporting service is used by default. Browsers post reports to the site REST endpoint registered by the plugin.

Why do violations appear in report-only mode?

That is expected. Report-only mode exists to show what would be blocked if the policy were enforced.

Why do I see connect-src violations after static scanning?

Static scans cannot reliably see runtime fetch, XHR, WebSocket, or EventSource calls. These are learned from real browser reports.

Why are some blocked URIs ignored as source candidates?

The plugin ignores non-host values such as inline, eval, data, blob, and about. Those are not normal host-source approvals.

Can violation reports be spoofed?

Browser reports are client-generated and should not be treated as trusted user input. The plugin validates content type, silently discards cross-origin document URIs, deduplicates reports, and rate-limits storage.

What does occurrence count mean?

It shows how often the same violation fingerprint has been seen. A high count often means a real repeated site behaviour rather than a single test.

Compatibility

WordPress, plugins, themes, and caching.

Is it compatible with caching plugins?

Usually, but nonce-based pages should not be served from a full-page cache unless the cache is nonce-aware or appropriate exclusions are configured.

Can it work with tag managers?

It can, but tag managers often load other scripts dynamically. Use report-only mode and approve only the hosts that are genuinely required.

Can it work with payment providers?

Yes, where the required scripts, frames, connections, and form actions are approved. Test checkout in report-only before enforcing.

Can it work with page builders?

Often, but page builders can introduce inline scripts, inline styles, external embeds, and runtime calls. Test representative pages before enforcing.

Does it add 'unsafe-inline'?

No. The nonce workflow is intended to avoid relying on 'unsafe-inline'.

What about older browsers?

Some reporting and CSP3 behaviour varies by browser. The plugin emits both modern Reporting-Endpoints and legacy Report-To headers for reporting compatibility.

WordPress.org build

Local features and guideline posture.

Does the plugin work without payment?

Yes. All features shipped in the WordPress.org package are available locally without payment or remote entitlement checks.

Does the plugin use a custom update endpoint?

No. The WordPress.org package does not poll a third-party update manifest or install updates from outside the WordPress.org update system.

Does the plugin contact a licensing service?

No. The WordPress.org package does not contact third-party services for licensing, checkout, telemetry, or remote product configuration.

Are CSP reports sent to VCNS or another provider?

No. Browser CSP reports are posted back to this WordPress site's own REST endpoint and stored locally.

Are new CSP sources applied automatically?

No. New sources remain pending until an administrator approves them.

Does manual automation mode change policy on its own?

No. Automation configuration defaults to manual mode on install and upgrade.

Privacy

What data exists and where.

What is stored locally?

Policy profiles, source inventory, hash inventory, violation reports, scan logs, audit log entries, policy versions, and policy change decisions are stored in the WordPress database.

Does normal CSP operation require telemetry?

No general telemetry is required for normal CSP operation. Browser reports are posted back to the site endpoint by default.

What leaves the site?

The WordPress.org package keeps CSP operation local. It does not contact third-party services for plugin updates, licensing, checkout, telemetry, or remote product configuration.

Troubleshooting

Fast answers during incidents.

My site broke after I enabled enforce mode. What should I do?

Return the affected surface to report-only. Review the Violations tab. Approve only the required sources or hashes. Retest before enabling enforce mode again.

I cannot access wp-admin after changing CSP. What now?

Use an alternative administrator route or database access to return the admin surface to report-only. Then review recent violations and policy changes.

The manual scan finds no sources. Is that wrong?

Not necessarily. Static scans see rendered HTML and linked CSS. Runtime-only activity may appear only through browser violation reports.

How do I report a security vulnerability?

Do not report suspected vulnerabilities in public issues. Use the repository security policy and the private reporting route described there.
