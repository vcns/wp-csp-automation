Quick Start

How to get from zero to report-only safely.

Use this sequence on a clean site or staging environment before you attempt production enforcement.

  1. Install and activate the plugin on WordPress 6.4+ running PHP 8.1+.
  2. Open CSP Manager settings and verify the site can run a manual scan.
  3. Run the first scan and review the source inventory for external origins.
  4. Stay in report-only mode while monitoring violations and approving only required sources.
  5. Promote one surface at a time to enforce mode after the inventory is stable.

Installation

Both the standard and manual install paths.

System Requirements

  • WordPress 6.4 or higher
  • PHP 8.1 or higher
  • php.ini 'sodium' available for Ed25519 signature verification

Normal Installation

  • Go to Plugins, then Add New Plugin.
  • Search for WP CSP Automation Manager.
  • Install and activate the plugin.

Manual Installation

  • Download the plugin ZIP package.
  • Open Plugins, then Add New Plugin, then Upload Plugin.
  • Select the ZIP file and install it.
  • Activate the plugin after upload.

Activation

  • Open CSP Manager settings.
  • Run the first scan.
  • Review source inventory and violations.
  • Keep each surface in report-only until ready.

Premium Activation

How entitlement activation actually works.

Premium access is activated through Stripe Checkout and a verified webhook, not from a browser redirect alone.

  1. Configure the Stripe mode, publishable key, secret key, and webhook secret in the plugin settings.
  2. Ensure Stripe is configured to send the supported checkout session webhook events to the plugin REST endpoint.
  3. Start checkout from the plugin premium screen.
  4. Complete the one-time payment in Stripe Checkout.
  5. Wait for the verified webhook to grant the entitlement locally on the site.

What Stripe is not used for

Free features do not require Stripe. The plugin does not perform per-request remote licence validation during normal runtime.

Entitlement scope

The stored entitlement is tied to the current site's identity, so domain changes affect licence matching.

Stripe and Privacy

What leaves the site, what stays local, and what must never be exposed.

Stripe usage

Stripe is used only for premium checkout and webhook processing. The plugin creates one-time Checkout Sessions and verifies webhook signatures before granting entitlements.

Local data storage

Violation reports, source inventory, hashes, scan logs, and entitlements are stored in the local WordPress database.

Remote config

The plugin fetches a signed JSON config document from an HTTPS endpoint discovered through DNS. That config must contain public product metadata only, never secrets.

Secrets boundary

Stripe secret keys, webhook secrets, and private signing keys must never be placed in DNS records, public config payloads, or browser-delivered code.

Privacy model

No general telemetry is intended as part of normal runtime. Outbound requests are limited to Stripe during purchase flow and your remote config endpoint for public signed metadata.

Verification model

Remote config is signature-verified with Ed25519 when `sodium` is available. Stripe webhook events are verified before any entitlement change is made.

Operating Model

What this plugin is designed to do.

  • Generate and emit per-surface CSP headers for frontend, admin, login, and API requests.
  • Inject nonces into WordPress-managed scripts without depending on unsafe inline policy shortcuts.
  • Discover third-party origins, store them as pending inventory, and require explicit operator approval.
  • Accept browser violation reports locally and use them to tighten allowlists over time.
  • Handle premium feature entitlement locally after verified Stripe webhook delivery.
  • Distribute public product metadata through signed remote config instead of hardcoding it in releases.

Safe rollout principle

Every surface begins in report-only mode. You should promote to enforce mode only after reviewing violations and approving the required sources or hashes.

Security principle

Entitlement grant decisions come from verified webhooks, not browser redirects. Remote config contains no secrets. Source discovery does not auto-approve anything.

Setup Guides

Admin and operator checklists.

Site admin checklist

  • Confirm WordPress and PHP version compatibility.
  • Install and activate the plugin.
  • Run a manual scan.
  • Review and approve only legitimate sources.
  • Monitor the violations table before enforcing.

Stripe operator checklist

  • Create products and one-time prices.
  • Configure webhook events for checkout session outcomes.
  • Set test and live credentials correctly in plugin settings.
  • Validate duplicate webhook idempotency.
  • Keep secrets out of docs, code, and client-side output.

Remote config checklist

  • Publish the DNS TXT record in the documented format.
  • Host signed JSON over HTTPS.
  • Store the Ed25519 private key outside WordPress.
  • Replace the placeholder public key in the plugin before release.
  • Test config refresh and stale-cache fallback behaviour.

Release Flow

How the repository is expected to ship.

The repository uses protected `development` and `main` branches with short-lived `feature/*`, `fix/*`, and `release/*` branches. GitHub Actions handles linting, branch policy enforcement, packaging, security scanning, and WordPress.org deployment.

  • Feature and fix branches merge into `development` by PR only.
  • Release branches are cut from `development` for final stabilization.
  • `main` is the source of tagged releases and WordPress.org deployment.
  • Documentation changes under `docs/` are now the public GitHub Pages source.

CI

PHP lint, PHPCS, Semgrep, secret scanning, and build artifact generation.

PR policy

Source-branch enforcement for pull requests into `development` and `main`.

Pages publish

GitHub Pages serves the `main:/docs` root path for the public help site.

WordPress.org deploy

Tag-driven deploy using environment-scoped SVN credentials.

FAQ

Questions people will ask before trusting this on a live site.

Will this break my site?

No. Every surface starts in report-only mode. The browser reports violations but does not block any resources. You can review violations and approve sources before ever enabling enforce mode.

What is the promotion gate?

The plugin blocks switching to enforce mode unless at least one source or hash has been approved for that surface, and no recent unresolved violation has been reported. This two-condition check reduces the risk of locking users out with an over-strict policy.

Should I approve every discovered source?

No. Approve only sources that are genuinely required for the site to function. Deny anything that looks unknown, unnecessary, or tracking-related.

Where are violation reports stored?

They are stored locally in the WordPress database in the site's csp_violation_reports table. Nothing is sent to an external reporting service by default; reports are deduplicated by fingerprint. Old reports are purged automatically: by default, rows older than 90 days are deleted after every daily scan. This threshold is configurable in CSP Manager → Settings (wp_csp_violation_retention_days); set it to 0 to keep reports indefinitely.

Does this plugin require Stripe?

No. Free features work without Stripe credentials. Stripe is only required if you want to use the premium purchase and entitlement flow.

Is the premium licence tied to one WordPress install?

Yes. The entitlement is bound to a stable hash of the site URL, so moving to a different domain changes the site identity and affects licence matching.

What PHP extensions are required?

The plugin relies on `json` and `hash`, which are standard in PHP 8.x, and `sodium` for Ed25519 signature verification of the remote config. If `sodium` is unavailable, the plugin logs a warning and continues in degraded verification mode.

Is this compatible with caching plugins?

Usually, but nonce-based pages should not be served from a full-page cache unless nonce-aware exclusions are in place. The plugin does not configure those cache exclusions for you.

Can I enforce a strict CSP on wp-admin?

With caveats. WordPress core Trac #59446 is unresolved: some core admin screens and bundled themes emit inline scripts outside the WordPress script API, so strict nonce-based enforcement can break parts of the admin UI. The plugin surfaces a warning notice when the admin surface is set to enforce mode. Monitor violation reports carefully before committing to enforcement on this surface, and keep a report-only fallback path available.

I switched to Enforce mode and my site broke. What do I do?

Switch the affected surface back to report-only, review the violation entries for blocked resources, approve the required sources or hashes, and then test enforce mode again.

How do I report a security vulnerability?

Do not report vulnerabilities in public issues. Follow the repository security policy and use the private reporting process documented there.

Support

Where to look when something is wrong.

  • Use the dashboard violation table when a surface fails in enforce mode.
  • Use the source inventory before broadening allowlists manually.
  • Use the scan log when discovery results change unexpectedly.
  • Use Stripe delivery logs and processed event rows when premium grants are missing.
  • Use audit log warnings when remote config refresh fails.

Security reporting

Do not file public vulnerabilities in issues or discussion threads. Follow the repository security policy and use the private reporting channel described there.
