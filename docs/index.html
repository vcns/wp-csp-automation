Start here

What the product is for.

WP CSP Automation Manager helps a WordPress administrator introduce a stricter Content Security Policy without immediately blocking live site resources. It does this by keeping each site surface in report-only mode first, collecting browser violation reports, discovering external sources, and turning those observations into reviewable policy proposals.

The plugin is not a one-click security switch. It is a controlled workflow. The administrator still decides which third-party hosts, scripts, frames, fonts, images, connections, and forms are legitimate for the site.

Best first use

Use it on staging first. Run a scan, browse representative pages, review the source inventory, and keep report-only enabled until the violations table is quiet.

Do not start in enforce mode

Enforce mode can block scripts, styles, frames, fonts, media, and API calls. Move one surface at a time only after the dashboard evidence supports the change.

How it works

The operating model.

1. Detect the surface

Requests are treated as frontend, admin, login, or API. Each surface has its own policy profile and its own mode.

2. Emit a policy

The plugin builds a CSP header from strict defaults, approved hosts, approved inline hashes, and a per-request nonce.

3. Learn from evidence

Static scans find many external assets. Browser violation reports identify runtime activity that static HTML cannot reliably reveal.

4. Require approval

Discovered sources are proposed as pending changes. They are not included in the live policy until an administrator approves them.

5. Record decisions

Approvals, rejections, and reversions are recorded in the policy change ledger. Rejected or reverted fingerprints suppress repeated automatic proposals.

6. Promote safely

The promotion gate blocks enforce mode until the surface has approved inventory and no recent violation activity inside the configured window.

Learning curve

What users need to learn, in order.

1

Basic CSP terms

Learn what a surface, directive, source, nonce, report-only policy, enforce policy, and violation report mean.

2

Dashboard reading

Use Profiles for mode, Source Inventory for pending hosts, Policy Changes for the decision ledger, Violations for browser evidence, and Scan Log for crawl history.

3

Approval judgement

Approve only sources with a clear business or functional purpose. Treat script, connection, frame, form, and worker changes as higher risk.

4

Report-only evidence

Browse real user journeys while report-only is active. Checkout, login, forms, embedded media, maps, analytics, consent tools, and page builders often reveal missing sources.

5

Controlled enforcement

Enforce the lowest-risk surface first, monitor violations after the change, and keep a rollback path to report-only.

Dashboard

Where to look in the WordPress admin.

Open CSP Manager in the WordPress admin. The main dashboard is split into tabs that match the product workflow.

  • Profiles: see each surface and switch between report-only, enforce, and disabled.
  • Source Inventory: review discovered hosts, risk, state, evidence count, and last-seen time.
  • Policy Changes: review the audit trail for approvals, rejections, reversions, and suppressions.
  • Violations: inspect browser-reported blocked resources and occurrence counts.
  • Scan Log: check manual and scheduled scan results.

Manual scan

Use Run Manual Scan after installing the plugin, changing a theme, adding a plugin, altering a checkout flow, or adding third-party content.

Scheduled scan

The plugin schedules a daily scan. Notification email is used when a scheduled scan detects a policy change.

Approval decisions

How to decide what belongs in the policy.

Approve

Approve a source when it is required, expected, narrow, and explainable. Examples include the payment provider used by checkout, the font CDN used by the theme, or the analytics host already approved by the site owner.

Investigate

Investigate if the source appears only once, has no clear owner, is high risk, or relates to script, connection, form, frame, or worker behaviour. Ask which plugin, theme, or business process requires it.

Reject or revert

Reject unknown or unnecessary proposals. Revert a previously approved source when it is no longer required. Both actions suppress the same fingerprint from being proposed again automatically.

Premium

What premium changes.

The free tier provides the core report-only workflow, basic scan, basic dashboard, and violation endpoint. Premium features add multi-surface scanning, strict-dynamic, Trusted Types support, and additional analytics or export surfaces where available.

Payment processing is handled by the remote licensing service. Stripe secret material does not need to be stored in the WordPress installation.

Activation model

Premium access is not granted by a browser redirect alone. It depends on the remote licensing service and verified payment processing before the local entitlement is considered active.

FAQ preview

The questions most administrators ask first.

Will report-only mode block anything?

No. Report-only mode asks the browser to report would-be violations. It does not block the resource. This is the safest way to learn what a live WordPress site actually needs.

Can enforce mode break a site?

Yes. Enforce mode can block required resources. Use it only after reviewing violations and approving required sources.

Operating Model

What this plugin is designed to do.

  • Generate and emit per-surface CSP headers for frontend, admin, login, and API requests.
  • Inject nonces into WordPress-managed scripts without depending on unsafe inline policy shortcuts.
  • Discover third-party origins, store them as pending inventory, and require explicit operator approval.
  • Accept browser violation reports locally and use them to tighten allowlists over time.
  • Raise higher-risk source proposals for administrator review before they can affect emitted headers.
  • Keep an append-only decision ledger so approvals, rejections, reversions, and suppressions remain auditable.
  • Handle premium feature entitlement locally after verified Stripe webhook delivery.
  • Distribute public product metadata through signed remote config instead of hardcoding it in releases.

Safe rollout principle

Every surface begins in report-only mode. You should promote to enforce mode only after reviewing violations and approving the required sources or hashes.

Security principle

Entitlement grant decisions come from verified webhooks, not browser redirects. Remote config contains no secrets. Source discovery does not auto-approve anything, and reverted sources are not automatically proposed again.

Setup Guides

Admin and operator checklists.

Site admin checklist

  • Confirm WordPress and PHP version compatibility.
  • Install and activate the plugin.
  • Run a manual scan.
  • Review and approve only legitimate sources.
  • Monitor the violations table before enforcing.

Stripe operator checklist

  • Create products and one-time prices.
  • Configure webhook events for checkout session outcomes.
  • Set test and live credentials correctly in plugin settings.
  • Validate duplicate webhook idempotency.
  • Keep secrets out of docs, code, and client-side output.

Remote config checklist

  • Publish the DNS TXT record in the documented format.
  • Host signed JSON over HTTPS.
  • Store the Ed25519 private key outside WordPress.
  • Replace the placeholder public key in the plugin before release.
  • Test config refresh and stale-cache fallback behaviour.

Release Flow

How the repository is expected to ship.

The repository uses protected `development` and `main` branches with short-lived `feature/*`, `fix/*`, and `release/*` branches. GitHub Actions handles linting, branch policy enforcement, packaging, security scanning, and WordPress.org deployment.

  • Feature and fix branches merge into `development` by PR only.
  • Release branches are cut from `development` for final stabilization.
  • `main` is the source of tagged releases and WordPress.org deployment.
  • Documentation changes under `docs/` are now the public GitHub Pages source.

CI

PHP lint, PHPCS, Semgrep, secret scanning, and build artifact generation.

PR policy

Source-branch enforcement for pull requests into `development` and `main`.

Pages publish

GitHub Pages serves the `main:/docs` root path and the latest stable update manifest.

WordPress.org deploy

Tag-driven deploy using environment-scoped SVN credentials.

FAQ

Questions people will ask before trusting this on a live site.

Will this break my site?

No. Every surface starts in report-only mode. The browser reports violations but does not block any resources. You can review violations and approve sources before ever enabling enforce mode.

What is the promotion gate?

The plugin blocks switching to enforce mode unless at least one source or hash has been approved for that surface, and no recent unresolved violation has been reported. This two-condition check reduces the risk of locking users out with an over-strict policy.
